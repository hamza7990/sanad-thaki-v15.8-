# تقرير v14.2.3-staging-repair-lock

هذه نسخة إصلاح مباشرة بعد فحص v14.2.2، وليست توسعة منتج.

## الإصلاحات المنفذة

1. إصلاح فشل SQL في قاعدة بيانات نظيفة عبر نقل تعديلات `support_tickets` بعد إنشاء الجدول.
2. إضافة `apps/api/scripts/migrate-db.mjs` وتشغيله تلقائيًا قبل بدء API.
3. نسخ ملفات SQL إلى `apps/api/migrations` حتى تكون داخل Docker build context.
4. تعديل Docker Compose لتحميل مجلد `infra/postgres` كاملًا في تهيئة PostgreSQL.
5. إعادة زر `رفع/قراءة فاتورة` ليستخدم endpoint حقيقي: `POST /invoices/read-file`.
6. endpoint القراءة يرفض أنواع الملفات غير المسموحة، يحدد حد 8MB، يسجل Audit Log، ويعيد بيانات مستخرجة مع `confidence` و`needsManualReview`.
7. نقل حد المستخدمين إلى إنشاء المستخدم فقط وعدم منع عرض قائمة المستخدمين عند الوصول للحد.
8. ضبط التقارير المتقدمة لتتطلب باقة الاحترافية من الخلفية والواجهة.
9. إزالة عبارة `داشبورد أدمن سند ذكي` من الواجهة.
10. تثبيت حد واتساب الاحترافية على 800 رسالة شهريًا.

## ملاحظات أمان وتشغيل

- لا تستخدم النسخة مع بيانات حقيقية قبل فحص الأمن السيبراني الخارجي وعزل الشركات والنسخ الاحتياطي.
- قراءة الفاتورة لا تعتمد ولا تقفل الفاتورة تلقائيًا؛ المحاسب يراجع البيانات ويحفظها ثم يرسلها للمراجعة.
- عند عدم توفر `OPENAI_API_KEY` يعمل استخراج احتياطي منخفض الثقة، ويحوّل المستخدم للمراجعة اليدوية.
