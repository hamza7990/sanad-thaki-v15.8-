# تقرير v14.2.10 — Async Invoice Queue Lock

تمت إضافة طابور معالجة خلفي للفواتير مع الحفاظ على قلب سند ذكي والحماية المعتمدة.

## ما تغير
- مسار رفع الفاتورة `/invoices/read-file` أصبح يعيد استجابة فورية `202 queued` بدل انتظار OCR/AI.
- حفظ ملف الفاتورة مشفرًا بمفتاح الشركة داخل `invoice_processing_jobs`.
- Worker خلفي يسحب المهام ويعالجها: جلسة AI معزولة → OpenCV hook → QR/AI/heuristic extraction → شرط 95% → تحديث الحالة.
- حالات المعالجة: `QUEUED`, `PROCESSING`, `PASSED`, `PENDING_REVIEW`, `FAILED`.
- API للاستعلام عن حالة المهمة: `/invoices/jobs/:jobId`.
- تسجيل استهلاك مستقل لكل شركة: `invoice_queued` و `invoice_processed_background`.
- كل عمليات القراءة تتم داخل `withTenant(companyId)` وبالتالي تحافظ على Database-per-Tenant أو RLS fallback حسب إعدادات البيئة.

## حماية القلب
- لم يتم تغيير صلاحيات المحاسب أو المدير المالي أو أدمن الشركة.
- لم يتم تغيير منطق الاعتماد المالي.
- لم يتم قبول `companyId` من العميل.
- الملفات المؤقتة الخاصة بالـ AI تبقى داخل جلسة tenant-scoped temp session ويتم حذفها بعد المعالجة.

## شرط القبول
- إذا اكتملت الحقول ووصلت الثقة إلى 95% أو أعلى: `PASSED`.
- إذا نقص حقل أو الثقة أقل من 95%: `PENDING_REVIEW` للمراجعة اليدوية.
- إذا فشلت المعالجة بعد المحاولات: `FAILED` مع تسجيل الخطأ.

## ملاحظة إنتاجية
الطابور الحالي DB-backed مناسب للنسخة الحالية بدون إضافة Redis. عند التوسع العالي يمكن نقل نفس المنطق إلى BullMQ/Redis كعامل خارجي بدون تغيير واجهات المستخدم أو قلب النظام.
