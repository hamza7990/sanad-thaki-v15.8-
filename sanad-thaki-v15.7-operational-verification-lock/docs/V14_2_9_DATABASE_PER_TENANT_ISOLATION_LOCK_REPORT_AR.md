# تقرير v14.2.9 — Database-per-Tenant Isolation Lock

تم تنفيذ طبقة عزل خلفية جديدة مع الحفاظ على قلب سند ذكي وواجهاته الحالية.

## ما تم إضافته

1. **Dynamic DB Routing**
   - `withTenant()` يوجه العمليات عبر `tenant-db-router.js`.
   - يمكن تعريف قاعدة مستقلة لكل شركة عبر `TENANT_DATABASE_URLS_JSON` أو متغيرات `TENANT_DB_URL_*`.
   - عند تفعيل `REQUIRE_DATABASE_PER_TENANT=true` يرفض النظام تشغيل شركة لا تملك قاعدة مستقلة.

2. **Encryption Key per Tenant**
   - إضافة `tenant-crypto.js` بتشفير AES-256-GCM لكل شركة.
   - تخزين نسخة مشفرة موازية لبيانات الفاتورة في `encrypted_payload` مع بقاء الحقول الأساسية لخدمة الواجهة الحالية.
   - في الإنتاج يجب استخدام KMS حقيقي وربط `TENANT_KMS_KEYS_JSON`/مزود KMS.

3. **AI Session Isolation**
   - كل قراءة فاتورة تعمل داخل جلسة مؤقتة معزولة لكل شركة.
   - يتم حذف الملفات المؤقتة تلقائيًا بعد انتهاء القراءة.
   - مسار OpenCV الخلفي محفوظ كـ hook: adaptive thresholding / deskewing / noise reduction.

4. **Tenant Usage Tracker**
   - جدول `tenant_usage_events` مع RLS/Force RLS.
   - تسجيل قراءات OCR، الفواتير اليدوية، مجموعة الفواتير، وفواتير الربط المحاسبي.

## ما لم يتغير

- قلب الفواتير والمحاسب والمراجعة المالية.
- RBAC/SOD.
- عزل `companyId` الحالي.
- واجهة المستخدم الأساسية.
- واتساب/البنك/التقارير إلا من ناحية عدم كسرها.

## ملاحظة إنتاجية مهمة

هذه بنية قوية جاهزة للاختبار، لكنها لا تعني اعتمادًا إنتاجيًا 100% قبل:
- تشغيل قواعد Tenant مستقلة فعليًا.
- تفعيل KMS حقيقي.
- اختبار شركة A ضد شركة B.
- مراجعة أمنية خارجية.
