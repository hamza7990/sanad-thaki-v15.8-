# تقرير المرحلة — v14.0.0 Commercial Full UI Candidate

## ما تم إنجازه

تم تجهيز نسخة تجارية مرشحة بواجهة مستخدم عربية كاملة تعمل من المتصفح عبر نفس خدمة API. النسخة تشمل شاشة دخول، تهيئة أولية لحساب مدير النظام، لوحة تحكم، إدارة بيانات الشركة، إدارة المستخدمين، فواتير العملاء، مراجعة واعتماد المدير المالي، واتساب العملاء بعد الاعتماد، عمليات البنك، مطابقة السداد، التقارير، الدعم الفني، وسجل التدقيق.

## ما تغير

أضيفت واجهة `apps/api/public`، وتم تعديل الخادم ليخدم `/login` و`/dashboard` و`/` من نفس التطبيق. تم إصلاح مسار الدخول تحت RLS باستخدام سياق داخلي مؤقت `app.login_lookup` بدل فتح قراءة المستخدمين لكل الاستعلامات. أضيفت جداول البنك والمطابقة ورسائل واتساب. أضيفت واجهات وإجراءات حقيقية مرتبطة بالـ API وليست صورًا أو واجهات وهمية.

## الأثر الأمني

تم الحفاظ على عزل الشركات عبر `company_id` وRLS، مع منع إدخال `companyId` من العميل. صلاحيات الأدوار بقيت مفصولة: الأدمن لإدارة النظام والشركة والمستخدمين، المدير المالي للاعتماد والمطابقة والتقارير، والمحاسب لإدخال الفواتير وتثبيتها وإرسال واتساب بعد الاعتماد. PostgreSQL لا يُكشف كبورت خارجي في Docker Compose. لا تزال النسخة غير مصرح بها لبيانات عملاء حقيقية قبل الفحص الأمني الخارجي.

## الفحوصات المنفذة

تم تنفيذ فحص نحوي لملفات JavaScript الأساسية، وفحص أمني/بنيوي مختصر، وفحص إصدار للتحقق من وجود ملفات الواجهة، RLS، Docker، وملفات التشغيل. لم يتم تنفيذ اختبار اختراق خارجي داخل هذه المرحلة.

## المخاطر المتبقية

يلزم اختبار Staging حقيقي بعد الرفع: إنشاء الأدمن الأول، إضافة محاسب ومدير مالي، إدخال فاتورة، تثبيتها، اعتمادها، إرسال واتساب، إضافة عملية بنك، تشغيل المطابقة، واعتماد المطابقة. كما يلزم فحص عزل شركة A/B، وفحص backup/restore، ومراجعة أمنية خارجية قبل أي بيانات حقيقية.

## قرار القبول

هذه النسخة تصلح كمرشح واجهة تجارية متكاملة للفحص على Staging وتسليمها لمختص النشر والأمن السيبراني. لا تعتبر Production نهائية ولا تُستخدم تجاريًا مع بيانات حقيقية إلا بعد اجتياز بوابات الأمن والنسخ الاحتياطي والعزل.

## الخطوة التالية
