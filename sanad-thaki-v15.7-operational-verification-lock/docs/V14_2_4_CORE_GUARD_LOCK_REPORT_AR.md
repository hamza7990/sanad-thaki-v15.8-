# تقرير إقفال v14.2.4 — Core Guard Lock

## القرار
تم إنشاء نسخة `v14.2.4-core-guard-lock` كنسخة إتقان وحماية لقلب البرنامج، مبنية على `v14.2.3-staging-repair-lock` دون فتح ميزات جديدة خارج نطاق الإقفال.

## سبب الإصدار
ظهر في النسخ السابقة خطر Regression أدى إلى سقوط أو ضعف مسار قراءة الفواتير أثناء تعديلات العزل والصلاحيات والواجهة. لذلك تمت إضافة حارس برمجي يمنع اعتماد أي نسخة مستقبلية إذا اختفى قلب البرنامج أو تحوّل زر أساسي إلى زر شكلي.

## ما تم إنجازه
- إضافة ملف `CORE_PROTECTION_MANIFEST.json` لتوثيق قلب سند ذكي غير القابل للكسر.
- إضافة سكربت `scripts/core-regression-guard.mjs` لفحص المسارات الأساسية قبل اعتماد أي نسخة.
- ربط حارس القلب مع فحص GitHub Actions ومع سكربتات Smoke Test.
- إضافة أوامر تشغيل مباشرة لحارس القلب على ويندوز ولينكس.
- تحسين واجهة المحاسب بحيث لا يظهر زر إرسال واتساب داخل صفحة الفواتير إذا كانت الباقة لا تسمح بالواتساب، حتى لو كانت الفاتورة معتمدة.
- إزالة بقايا دالة عرض الشروط والسياسات داخل التطبيق بعد الدخول، مع بقاء الصفحات العامة خارج النظام.
- تحديث الإصدار والـ Manifest والـ Changelog والـ README.

## قلب البرنامج المحمي
- رفع/قراءة الفاتورة.
- التصحيح اليدوي.
- إرسال الفاتورة للمراجعة.
- اعتماد/رفض المدير المالي.
- واتساب بعد اعتماد المدير المالي وبحسب الباقة فقط.
- إدخال/قراءة عمليات البنك.
- مطابقة السداد.
- عزل الشركات.
- صلاحيات الأدوار.
- حدود الباقات.
- Audit Log.
- الدعم الفني المعزول حسب الشركة.
- النسخ الاحتياطي والاستعادة.
- تسجيل الدخول والجلسات.

## أثر الأمان
لا يفتح هذا الإصدار ميزات جديدة. أثره الأمني إيجابي لأنه يحوّل حماية المسارات الأساسية إلى Release Gate آلي. لا تزال النسخة مخصصة لـ Staging فقط، وليست لبيانات عملاء حقيقية قبل الفحص الأمني الخارجي، واختبار عزل الشركات، واختبار النسخ الاحتياطي والاستعادة.

## الفحوصات المطلوبة لقبول النسخة
يجب أن تنجح الأوامر التالية قبل أي رفع أو اعتماد:

```bash
node --check apps/api/src/server.js
node --check apps/api/public/app.js
node scripts/security-check.mjs
node scripts/release-check.mjs
node scripts/core-regression-guard.mjs
```

أو تشغيل:

```bash
./RUN-FINAL-SMOKE-TEST-LINUX.sh
```

على ويندوز:

```cmd
RUN-FINAL-SMOKE-TEST-WINDOWS.cmd
```

## شروط الرفض التلقائي
ترفض أي نسخة إذا حدث أي مما يلي:
- اختفاء `/invoices/read-file`.
- وجود زر رفع/قراءة فاتورة بلا API حقيقي.
- اختفاء التصحيح اليدوي أو إرسال المراجعة.
- عمل واتساب قبل اعتماد المدير المالي أو خارج استحقاق الباقة.
- نقل `userLimitGuard` إلى عرض المستخدمين بدل إنشاء المستخدمين.
- كسر عزل `company_id` في أي مسار مالي أو دعم أو تدقيق.
- إعطاء المحاسب صلاحيات بنك/مطابقة/اعتماد/إدارة مستخدمين.
- إعطاء أدمن سند ذكي صلاحيات مالية داخل شركات العملاء.

## قرار القفل
`v14.2.4-core-guard-lock` هي نسخة Staging متقنة لحماية القلب، وليست Production لبيانات العملاء الحقيقية. التطوير الوظيفي يبقى مغلقًا، وأي تعديل لاحق يجب أن ينجح في حارس القلب قبل تسميته نسخة جاهزة.
